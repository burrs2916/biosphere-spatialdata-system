# 空间数据驱动系统 - 多数据源集成设计方案

## 文档信息
- **创建日期**：2026-04-14
- **版本**：v2.0
- **状态**：实施中
- **设计目标**：在不破坏现有认证逻辑的前提下，实现多数据源接入能力，支持 HTTP/WebSocket/MQTT/TCP/数据库等多种数据源类型，实现组件与数据源的动态绑定。

---

## 一、设计原则
### 核心约束
✅ **零侵入**：完全不修改现有认证逻辑和代码，保证已实现功能的稳定性
✅ **非侵入式设计**：只制定规则，不强制用户使用，用户可自定义数据源API，安全由用户自行负责
✅ **低风险**：分阶段落地，每阶段产出可独立验证
✅ **向后兼容**：现有功能不受任何影响，新功能作为独立模块添加

### 核心理念
1. **认证 baseURL 本身就是数据源**：用户如需安全的数据访问，直接用认证后端的 API 即可，通过已缓存的 token、tenantId、userId 等信息直接调用
2. **独立的数据源管理**：专门的数据源管理功能，和认证系统完全独立。用户自己添加数据源，自己实现 API，系统不保证安全
3. **只制定规则**：提供新增功能、对返回数据进行动态解析识别、数据与组件联动展示

### 架构分层
```
┌─────────────────────────────────────────────┐
│            UI 组件层 (展示)                    │
│  地图组件 / 图表组件 / 列表组件 / 自定义组件     │
│  动态绑定数据源ID，自动订阅数据更新              │
└──────────────┬──────────────────────────────┘
               │ 动态绑定 (数据订阅)
┌──────────────▼──────────────────────────────┐
│            数据源层 (连接，独立模块)             │
│  • 多协议适配器：HTTP / WebSocket / MQTT / TCP / DB
│  • 数据获取策略：轮询 / 订阅 / 按需
│  • 数据映射与缓存管理（独立缓存，与认证隔离）     │
│  • 安全由用户自行负责                           │
└──────────────┬──────────────────────────────┘
               │ 可选：只读认证缓存（token等）
┌──────────────▼──────────────────────────────┐
│            认证层 (现有，完全不变)              │
│  • Token 管理 / 自动刷新 / 请求签名
│  • 用户信息缓存：tenantId, userId, accessToken
│  • 认证后端API本身也可作为安全数据源使用          │
└─────────────────────────────────────────────┘
```

## 二、阶段 1：核心数据源框架（HTTP协议优先）
**目标**：实现数据源管理基础能力，支持 HTTP 数据源的 CRUD、请求执行、数据解析、轮询策略。

### 1. 数据结构定义
```typescript
// src/types/dataSource.ts

export type DataSourceType = "http" | "websocket" | "mqtt" | "tcp" | "database";
export type DataSourceStatus = "active" | "inactive" | "error" | "connecting";
export type FetchStrategy = "poll" | "on-demand" | "subscribe";
export type HttpMethod = "GET" | "POST" | "PUT" | "DELETE";

export interface DataSourceHeader {
  id: string;
  key: string;
  value: string;
  enabled: boolean;
}

export interface DataSourceParam {
  id: string;
  key: string;
  value: string;
  location: "query" | "body" | "header";
  enabled: boolean;
}

export interface DataSourceResponseMapping {
  id: string;
  sourcePath: string;
  targetKey: string;
  saveToCache: boolean;
}

export interface DataSource {
  id: string;
  name: string;
  type: DataSourceType;
  description?: string;
  enabled: boolean;
  status: DataSourceStatus;
  createdAt: string;
  updatedAt: string;

  // HTTP 连接配置
  connection: {
    url: string;
    method: HttpMethod;
    headers: DataSourceHeader[];
    params: DataSourceParam[];
    body?: string;
    timeout: number;
  };

  // 响应数据映射
  responseMapping: DataSourceResponseMapping[];

  // 获取策略
  strategy: {
    mode: FetchStrategy;
    interval?: number;
    autoStart: boolean;
  };

  // 绑定的组件ID列表
  boundComponents: string[];

  // 最后一次数据
  lastData?: any;
  lastFetchedAt?: string;
  lastError?: string;
}
```

## 五、与现有认证系统的关系
1. **完全独立**：数据源模块作为独立模块存在，不修改任何认证逻辑
2. **只读使用**：数据源模块只读取认证缓存中的信息（token、userId、tenantId等），不修改任何认证状态
3. **安全由用户负责**：如果需要安全的数据访问，直接使用认证后端的 API
4. **认证后端API本身也可作为数据源**：用户可以直接配置认证后端的API作为数据源，复用已缓存的认证信息
